Основы работы с Docker

Андрей Черных
19/11/2016 02:57

Docker — технология, позволяющая просто и быстро запускать приложение в контейнере, похожем на виртуальную машину. Основные черты, отличающие его от виртуальное машины — это легковесность, ресурсоёмкость и практически полная независимость от инфраструктуры.

С помощью одной команды мы можем развернуть на сервере WordPress, MySQL, Redis и многие другие популярные приложения. Нам абсолютно не нужно заботиться о дополнительных установках, все настройки и нужные сервисы уже содержаться в контейнере, остается его только запустить. Мы также можем разрабатывать свое приложение в контейнере и затем разворачивать его на сервере.

Docker основан на клиент-серверной архитектуре. Сервер отвечает за создание, запуск, остановку, распределение контейнеров, а клиент за интерфейс взаимодействия пользователя с сервером. Как правило клиент и сервер функционируют на одной машине, но при желании клиент может быть на другой.

Docker состоит из трех компонент: образы, контейнеры и реестры (репозитории).

Образ представляет из себя неизменяемый шаблон для создания контейнеров (например, образ может содержать Ubuntu, Nginx и MySQL). Для создания образов Docker использует Dockerfile, в котором содержится набор инструкций, формирующих образ. Каждый образ можно создавать на основе другого образа, базового.

Реестр — это своего рода хранилище для образов. Реестры делятся на публичные и приватные. DockerHub — самый большой публичных реестр, в котором можно найти образ с приложением почти для любой конфигурации сервера.

Контейнер запускается на основе образа и содержит все необходимое для его работы. Каждый контейнер запускается в изолированном окружении. На одной машине мы можем запустить несколько контейнеров и не беспокоится о том, что они будут мешать работе другу другу. Docker позволяет настраивать открытость контейнера к другим контейнерам и самой системе. С помощью Doсker мы также можем указывать максимальное количество потребляемых контейнером ресурсов.

В этом руководстве мы рассмотрим и изучим основные команды Docker для работы с образами и контейнерами,

Требования

  • Сервер Ubuntu 16.04 x64 (мы использовали следующую конфигурацию: Ubuntu 16.04 64 bit RAM 1024Mb и SSD 30 Гб);

  • Пользователь с root-правами.

Установка Docker

На этом шаге у нас есть два варианта

  • Использовать уже готовый сервер с Docker
  • Установить Docker самостоятельно


Vscale позволяет создать в разделе “приложения” уже готовый сервер на базе Ubuntu 16.04 с установленным Docker и приступить сразу к работе.

Для самостоятельной установки Docker на Ubuntu 16.04 необходимо выполнить нижеследующие команды.

Обновляем программные пакеты:

$ apt-get update

Добавляем GPG ключ:

$ apt-key adv --keyserver hkp://p80.pool.sks-keyservers.net:80 --recv-keys 58118E89F3A912897C070ADBF76221572C52609D

Устанавливаем пакет software-properties-common:

$ apt-get install software-properties-common

Добавляем репозиторий:

$ apt-add-repository 'deb https://apt.dockerproject.org/repo ubuntu-xenial main'

Вновь обновляем программные пакеты:

$ apt-get update

И затем устанавливаем Docker:

$ apt-get install -y docker-engine

После установки система автоматически запустит сервис Docker. Проверяем его статус с помощью команды:

$ service docker status



Docker установлен и готов к работе.

 

Основы работы с образами

Для просмотра всех установленных образов используется команда docker images. Выполним ее в командной строке:

$ docker images

Как видим, у нас ещё нет ни одного образа в системе. Давайте для примера найдем и установим образ с MySQL сервером.

Для поиска нужных образов, применяется команда docker search. Эта команда позволяет нам искать образы в публичном реестре DockerHub. Искать образы можно также на самом сайте.

Найдем образ с MySQL сервером.

$ docker search mysql


Поиск выдал нам несколько результатов. Каждый образ из списка в основном содержит MySQL, но со своими особенностями (например образ zabbix/zabbix-server-mysql содержит в себе систему мониторинга Zabbix на основе MySQL, что не совсем нам подходит). Более подробно о сборки можно прочитать на странице образа в DockerHub.

Мы будем устанавливать самый первый образ, который является официальным и содержит только MySQL. В реестре под одним именем могут лежать несколько образов с разными версиями приложения. В нашем примере, мы можем скачать образ с версиями MySQL: 5.5, 5.6, 5.7(latest) или 8.0.

Команда docker pull скачивает указанный образ. Получаем MySQL образ с версией 5.6

$ docker pull mysql:5.6

Дожидаемся, когда закончится установка и проверяем, что образ появился в списке:

$ docker images

Аналогично установим образ, который содержит внутри себя только Nginx.

$ docker pull nginx

Здесь мы не указывали явно версию, поэтому скачался образ с тегом latest (последняя версия).

Теперь у нас два скачанных образа:

Если есть необходимость удалить образ, используется команда docker rmi.

В качестве примера, удалим ранее установленный образ MySQL сервера.

$ docker rmi mysql:5.6

Тут важно указать версию, если этого не сделать, то Docker автоматически будет искать образ с тегом latest.

О том, как создавать свои образы с помощью Dockerfile можно прочитать в этой статье.

В данном разделе руководства мы научились искать, загружать, удалять и просматривать текущие образы. Переходим к работе с контейнерами.

 

Основы работы с контейнерами

Как уже было написано выше, для запуска приложений используются контейнеры. Так как мы удалили образ c MySQL, в системе остался только один образ с Nginx. Запустим его с помощью следующей команды:

$ docker run -d --name qa_nginx nginx

Команда docker run запускает контейнер на основе образа, имя которого указано в конце команды (в нашем случае nginx). Опция -d применяется для запуска контейнера в автономном режиме, а --name позволяет задавать имя контейнера (qa_nginx). Если не указывать опцию --name, то Docker присвоит контейнеру сгенерированное имя.


Для просмотра работающих контейнеров в Docker используется команда docker ps. Выполним её в консоли:

$ docker ps


Наш контейнер запущен и работает. В столбце PORTS можно посмотреть, какие порты используются внутри контейнера. Если мы сейчас зайдем на 80 порт, то ничего не увидим. Чтобы порты были доступны извне, их необходимо пробросить при запуске контейнера.

Удалим с помощью команды docker rm наш контейнер.

$ docker rm -f qa_nginx

Опция -f позволяет удалять запущенные контейнеры без их предварительной остановки.

Запустим новый контейнер с уже проброшенным 80 портом

$ docker run --name qa_nginx -d -p 80:80 nginx

Теперь при запросе к внешнему 80 порту, система будет автоматически перенаправлять запрос на 80 порт контейнера. Аналогично можно пробросить порт 443.

Проверим, что всё работает. Заходим на 80 порт нашей машины и видим страницу приветствия Nginx:

Связь контейнера с локальной файловой системой

Чтобы заменить какой-нибудь файл в контейнере, используется команда docker cp. Она копирует файл из локальной файловой системы в контейнер. В нашем примере, мы можем изменить конфигурационный файл nginx следующей командой:

$ docker cp qa_nginx:/etc/nginx/nginx.conf /tmp/nginx.conf

Где qa_nginx и /etc/nginx/nginx.conf - имя контейнера и путь к

заменяемому файлу внутри контейнера соответственно, а /tmp/nginx.conf путь до нового файла в текущей операционной системе.

Более подробно о настройках каждого контейнера, можно прочитать на официальной странице образа в DockerHub.

Docker также позволяет ссылаться на файлы и каталоги в локальной файловой системе с помощью опции -v при запуске контейнера.

Для демонстрации пробросим корневую директорию, в которой лежат ресурсы сайта nginx.

Переходим в папку tmp и создаем там директорию /nginx/content:

$ cd /tmp
$ mkdir -p nginx/content
$ cd nginx/content

Теперь создаем пустой html файл:

$ >index.html

Открываем его на редактирование:

$ nano index.html

И вставляет в него следующий текст:

<!DOCTYPE html>
<html>
<head>
    <title>Vscale example</title>
</head>
<body>
    <h1>Hellow</h1>
</body>
</html>

Нажимаем в редакторе сочетание клавиш Ctrl+x, соглашаемся с сохранением и нажимаем Enter.

Чтобы пробросить эту директорию в контейнер с nginx, необходимо удалить сначала старый контейнер

$ docker rm -f qa_nginx

И выполнить следующую команду:

$ docker run --name qa_nginx -v /tmp/nginx/content:/usr/share/nginx/html:ro -d -p 80:80 nginx

Заходим на 80 порт и видим, что главная страница поменялась. Так как мы не скопировали директорию, а сделали на неё ссылку, то изменения в локальной директории повлекут за собой изменения в контейнере.

Более подробно о настройке Nginx можно прочитать здесь.

Остановка и возобновление работы контейнера

Для остановки и возобновление работы контейнеров используются команды docker stop и docker start соответственно. Для просмотра всех контейнеров, а не только запущенных, команду docker ps необходимо запускать с опцией -a.

Остановим наш контейнер:

$ docker stop qa_nginx

Запустим новый контейнер на основе nginx:

$ docker run --name test_nginx -d nginx

И выполним команду:

$ docker ps -a

В столбце STATUS мы можем отслеживать текущее состояние контейнера.

Работа с терминалом

Docker также позволяет отправлять команды контейнеру, выполним:

$ docker run --name tty_nginx -it nginx /bin/bash

Здесь опция -it запускает в интерактивном режиме псевдоконсоль TTY, в которой мы можем работать напрямую с контейнером. В качестве примера установим в контейнере файловый менеджер mc:

$ apt-get update
$ apt-get install mc

Проверяем, что всё работает:

$ mc

Закрываем mc и выходим из консоли контейнера:

$ exit;

Если мы удалим наш контейнер и вновь его запустим, то все наши изменения, сделанные через консоль, исчезнут. Чтобы сохранить изменения используется команда docker commit:

$ docker commit 2404abeecf66 nginx_mc

Данная команда создаст новый образ nginx_mc, в котором будет уже установленный редактор mc. Здесь 2404abeecf66 - id контейнера, который можно узнать выполнив команду docker ps.

На официальном сайте Docker  опубликована подробная  документация с примерами, где вы можете более подробно прочитать об опциях рассмотренных выше команд.

Заключение

В этом руководстве мы познакомились с Docker и на примерах рассмотрели основные команды для работы с образами и контейнерами.
